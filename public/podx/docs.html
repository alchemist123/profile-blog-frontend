PodX Documentation

PodX is a distributed LLM inference cluster written in Rust. Install one binary on multiple devices, and they form a cluster that can run models too large for any single machine.

✅
OpenAI-compatible. PodX exposes /v1/chat/completions with streaming SSE. Any app that works with the OpenAI Python SDK, the Node.js client, or curl works with PodX — just change the base URL.

What PodX does

  • Discovers nearby nodes on your LAN via mDNS — zero configuration
  • Elects a coordinator based on hardware score (RAM × CPU × GPU)
  • Splits model transformer layers across all nodes (pipeline parallelism)
  • Routes inference requests through the pipeline over gRPC
  • Returns streaming tokens via an OpenAI-compatible HTTP API
  • Shows live cluster state in an embedded web dashboard

Installation

PodX is distributed as a single binary built from source with Cargo. You'll need the Rust toolchain and a few platform-specific build tools.

macOS — Apple Silicon & Intel

Tested on macOS 12 Monterey through macOS 15 Sequoia, on both M-series (arm64) and Intel (x86_64) Macs.

1. Install Rust

bash
# Install the Rust toolchain (includes rustc and cargo)
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
source ~/.cargo/env

# Verify
rustc --version   # should print: rustc 1.78.0 or later
cargo --version

2. Install build dependencies

bash
# Install Xcode Command Line Tools (includes clang, ar, ld)
xcode-select --install

# Install Homebrew (if not already installed)
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

# Install protobuf compiler (required for gRPC code generation)
brew install protobuf cmake

3. Clone and build

bash
git clone https://github.com/yourname/podx.git
cd podx

# CPU-only build (works on all Macs)
cargo build --release

# Apple Silicon — enable Metal GPU acceleration (M1/M2/M3/M4)
cargo build --release --features podx-inference/metal

# Verify the binary
./target/release/podx --version
💡
On Apple Silicon, always build with --features podx-inference/metal for 5–10× faster inference. Metal offloads all transformer layers to the GPU's Neural Engine.

4. (Optional) Install to PATH

bash
# Install to ~/.cargo/bin (already on PATH if you used rustup)
cargo install --path crates/podx-node

# Or copy manually
cp target/release/podx /usr/local/bin/podx

Quick Start

Get PodX running on a single machine in under two minutes.

1

Start the node

Run podx start. PodX detects your hardware, starts gRPC on port 9090 and HTTP on port 8080, and begins advertising on mDNS.

2

Open the dashboard

Navigate to http://localhost:8080/ui in your browser. You should see your node appear as Coordinator with its hardware details.

3

Load a model

From the Models page, enter the path to a local GGUF file and click Load. Or use the CLI: podx model load --file /path/to/model.gguf

4

Send your first request

Use the Chat page in the dashboard, or call the API directly with curl or any OpenAI-compatible client.

bash — start + first request
# Terminal 1 — start the node
./podx start

# Terminal 2 — load a model (adjust path)
./podx model load --file ~/models/llama-3.2-3b-instruct-q4_k_m.gguf

# Terminal 2 — send a chat completion request
curl http://localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model":"podx-default","messages":[{"role":"user","content":"Hello!"}],"stream":true}'

Single Node

bash
# Default ports (gRPC: 9090, HTTP: 8080)
./podx start

# Custom ports
./podx start --port 9091 --http-port 8081

# Limit inference threads
./podx start --threads 4

# Force this node to be coordinator
./podx start --force-coordinator
ℹ️
On a single node, PodX acts as both coordinator and worker. It handles requests end-to-end. The layer range will be L0–N (all layers).

Multi-Node Cluster

Run podx start on each machine on the same LAN. Nodes find each other via mDNS within seconds.

bash — Machine 1 (auto coordinator if highest score)
./podx start
# → Advertises on mDNS, waits for peers
bash — Machine 2 (discovers Machine 1 automatically)
./podx start
# → Discovers Machine 1 via mDNS, cluster forms
bash — Manual connect (Tailscale, VPN, different subnet)
# Connect directly by IP when mDNS doesn't reach
./podx start --connect 100.64.0.1:9090
./podx start --connect 192.168.2.50:9090

Distribute a model across all nodes

bash
# Install — computes shard map and pushes layers to each node
./podx model install --file ~/models/llama-3-70b-q4.gguf

# Check cluster status
./podx status

GPU Acceleration

PodX uses llama.cpp under the hood via the llama-cpp-2 Rust crate. GPU support is compiled in via Cargo feature flags — no runtime configuration needed.

PlatformFeature flagBuild commandDetection
Apple Silicon (M1–M4)podx-inference/metalcargo build --release --features podx-inference/metalsystem_profiler SPDisplaysDataType
NVIDIA (Linux/Windows)podx-inference/cudacargo build --release --features podx-inference/cudanvidia-smi
CPU (all platforms)none (default)cargo build --release—
💡
When GPU support is compiled in, PodX automatically passes n_gpu_layers=9999 to llama.cpp — offloading all transformer layers to the GPU. llama.cpp clamps this to the actual layer count, so the value is safe regardless of model size.

On startup, PodX logs the detected compute backend:

log output

INFO GPU: Metal detected, vram=10 GB (unified memory)
INFO Compute: Metal GPU (10 GB VRAM)
INFO Hardware score: 4300

Loading Models

PodX uses GGUF quantized models — the same format used by llama.cpp and Ollama. Download models from Hugging Face (look for repos with gguf in the name).

Load on local node

bash
# Load with default context (2048 tokens)
./podx model load --file /path/to/model.gguf

# Load with extended context
./podx model load --file /path/to/model.gguf --n-ctx 4096

# Limit to 4 inference threads
./podx model load --file /path/to/model.gguf --threads 4

Install across cluster

bash
# Distribute shards to all online nodes (run from any node)
./podx model install --file /path/to/llama-3-70b-q4_k_m.gguf

# The coordinator computes the shard map and pushes layers to each node
# You'll see progress for each node in the logs

Authentication

By default, PodX runs without authentication. Enable it by setting require_auth: true via the API or dashboard Settings page.

Generate an API key

bash
# Generate a Full-access key
./podx key generate --name my-app --scope Full

# Chat-only key (inference endpoints only)
./podx key generate --name frontend --scope Chat

# Read-only key (status endpoints only)
./podx key generate --name monitor --scope ReadOnly

# List all keys
./podx key list

# Revoke a key
./podx key revoke sk-podx-abc123...

Using a key

bash
curl http://localhost:8080/v1/chat/completions \
  -H "Authorization: Bearer sk-podx-your-key-here" \
  -H "Content-Type: application/json" \
  -d '{"model":"podx-default","messages":[{"role":"user","content":"Hi"}]}'
ScopeAllowed endpoints
FullAll endpoints — config, key management, inference, status
Chat/v1/chat/completions and inference routes only
ReadOnly/api/nodes, /api/metrics, /api/state only

CLI Reference

podx start

podx start [OPTIONS]
Start the PodX node. Detects hardware, starts gRPC and HTTP servers, advertises on mDNS, and joins or forms a cluster.
--port <n> gRPC listen port (default: 9090)
--http-port <n> HTTP listen port (default: 8080)
--connect <host:port> Connect to a specific peer (disables mDNS for that peer)
--force-coordinator Override election and become coordinator regardless of score
--threads <n> Number of inference threads (default: physical CPU cores)

podx model

podx model load --file <path> [OPTIONS]
Load a GGUF model file on the local node and make it active for inference.
--file <path> Path to the GGUF file (required)
--n-ctx <n> Context window size in tokens (default: 2048)
--threads <n> Inference threads for this model
podx model install --file <path> [OPTIONS]
Distribute model shards across all online cluster nodes. Computes the optimal shard map and pushes each node's layer range via gRPC.
podx model list
List all models known to the cluster with their metadata.
podx model info <model-id>
Show detailed metadata for a model: layer count, quantization, hidden dim, vocab size, file size, checksum.

podx node

podx node list
List all nodes in the cluster with their status, hardware, and assigned layer range.
podx node ping <node-id>
Send a heartbeat probe to a specific node and report its response time.
podx node kick <node-id>
Remove a node from the cluster state. The node will re-join on next heartbeat.
podx node promote <node-id>
Force a node to become coordinator by temporarily inflating its score.

podx key

podx key generate [--name <label>] [--scope <scope>]
Generate a new API key. Scope must be one of: Full, Chat, ReadOnly. The key is shown once — save it immediately.
podx key list
List all API keys (masked) with their name, scope, and creation date.
podx key revoke <key>
Permanently revoke an API key. Takes effect immediately on all nodes via gossip.

podx config

podx config show
Show the current cluster configuration (lb_strategy, heartbeat_ms, max_queue, require_auth).
podx config set <key> <value>
Update a cluster configuration value. Propagates to all nodes via gossip. Changes take effect immediately.

Coordinator Election

When a PodX cluster forms, one node becomes the coordinator. The coordinator receives all inference requests, maintains the shard map, and routes the pipeline.

Election is score-based and deterministic:

rust — scoring formula
fn score(hw: &HardwareInfo) -> u64 {
    hw.ram_gb * 100
    + hw.cpu_cores as u64 * 50
    + hw.gpu_vram_gb.unwrap_or(0) * 200
    + if hw.os == "macos" { 2000 } else { 0 }
}

Every node broadcasts its score via gossip. The highest score wins. Ties are broken by lexicographic node ID. If the coordinator goes offline, the cluster re-elects instantly without any operator action.

Load Balancing

StrategyBehaviorBest for
LeastLoadedRoute to node with fewest queued requestsGeneral use (default)
RoundRobinDistribute in strict rotationUniform hardware
LatencyAwarePrefer nodes with lowest recent avg latencyLatency-sensitive apps
WeightedProbability ∝ hardware scoreMixed hardware clusters
RandomRandom available nodeTesting, even distribution

Strategies can be changed live without restarting any node. The change propagates via gossip and takes effect on the next request.

Gossip Protocol

PodX uses a simple push gossip protocol to keep ClusterState consistent across all nodes.

  • Every node holds a complete, versioned copy of ClusterState
  • On a ticker (default: 3s), each node pushes its current state to all known peers via the Gossip gRPC RPC
  • A peer accepts an incoming state only if its version is higher than its local copy
  • State changes (node joins, model loads, config updates) propagate to the full cluster in O(hops × tick_interval)
  • No central server, no Raft, no Paxos — simple and crash-tolerant
